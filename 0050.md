## Dependently typed program transformations

*This post is jointly written with Shu-Ting Hsu.*

When writing Agda programs, pattern matching plays an important role in shaping the structure of a definition ‚ÄĒ that is, applying different pattern matching strategies to the same function leads to different program structures corresponding to different ways of performing case analyses.
In particular, when working with an indexed data type, a beginner may analyse the indices first but end up with a more complex program, and then they are usually taught that it can be more straightforward to analyse the data.
For example, consider this definition of `_‚Č§_` (which is a renamed version of `_‚Č§‚Ä≤_` from the standard library)
```
data _‚Č§_ : ‚Ąē ‚Üí ‚Ąē ‚Üí Set where
  base : m ‚Č° n ‚Üí m ‚Č§ n
  step : m ‚Č§ n ‚Üí m ‚Č§ suc n
```
and suppose that we want to weaken `suc m ‚Č§ n` (that is, `m < n`) to `m ‚Č§ n`:
```
wk : (m n : ‚Ąē) ‚Üí suc m ‚Č§ n ‚Üí m ‚Č§ n
wk m .(suc m) (base refl)    = step (base refl)
wk m .(suc n) (step {n} m<n) = step (wk m n m<n)
```
This definition performs a case analysis on the proof `suc m ‚Č§ n`.
To emphasise that the pattern matching on the second argument is not performed by the programmer but forced by unification, we use dot patterns explicitly (and therefore need to explicitly bind `n` in the `step` constructor in the second clause).
A beginner might be unfamiliar with the idea of induction on proofs though, and might write the following version that analyses the indices `m` and `n` first:
```
wk' : (m n : ‚Ąē) ‚Üí suc m ‚Č§ n ‚Üí m ‚Č§ n
wk'  zero    zero          m<n         = base refl
wk'  zero   (suc .zero)    (base refl) = step (base refl)
wk'  zero   (suc n)        (step m<n)  = step (wk' zero n m<n)
wk' (suc m)  zero          (base ())
wk' (suc m) (suc .(suc m)) (base refl) = step (base refl)
wk' (suc m) (suc n)        (step m<n)  = step (wk' (suc m) n m<n)
```
Here, `wk'` is finished with six cases, and is more complex than `wk`.
In all the cases except the first one,

* once the indices `m` and `n` have been fixed, the shape of `suc m ‚Č§ n` is already forced. (still need to do the matching explicitly)
  * the surrounding index patterns already determine which proof forms are possible
* `base refl` vs `step (base refl)` in the first case

Thus `wk` and `wk'` have different case analysis structures, even though extensionally they define the same function.
So, what is the connection between them? By transforming `wk'` step by step into `wk`, we can make those structural differences explicit.

### Transforming `wk'` to `wk`

The first step is to fully expand the two levels of case analyses in every clause.
In the case of `wk'`, only the first clause still needs to be expanded:
```
  wk‚āĀ : (m n : ‚Ąē) ‚Üí suc m ‚Č§ n ‚Üí m ‚Č§ n
  wk‚āĀ  zero    zero          (base _)    = base refl
  wk‚āĀ  zero   (suc .zero)    (base refl) = step (base refl)
  wk‚āĀ  zero   (suc n)        (step m<n)  = step (wk‚āĀ zero n m<n)
  wk‚āĀ (suc m)  zero          (base ())
  wk‚āĀ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wk‚āĀ (suc m) (suc n)        (step m<n)  = step (wk‚āĀ (suc m) n m<n)
```
Next, we reorder the clauses so that the clauses with the same patterns for  `suc m ‚Č§ n` are grouped together, to indicate that the case analysis on the indexed data `suc m ‚Č§ n` is now performed first (although the case tree generated by Agda won‚Äôt reflect that):
```
  wk‚āā : (m n : ‚Ąē) ‚Üí suc m ‚Č§ n ‚Üí m ‚Č§ n
  wk‚āā  zero    zero          (base _)    = base refl
  wk‚āā  zero   (suc .zero)    (base refl) = step (base refl)
  wk‚āā (suc m)  zero          (base ())
  wk‚āā (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wk‚āā  zero   (suc n)        (step m<n)  = step (wk‚āā  zero   n m<n)
  wk‚āā (suc m) (suc n)        (step m<n)  = step (wk‚āā (suc m) n m<n)
```
In the `step` clauses, the right-hand sides do not actually depend on whether the first argument is `zero` or `suc m` ‚ÄĒ the difference is only in the first arguments of the recursive calls, but what they both do is copying the first argument, regardless of whether that‚Äôs `zero` or `suc m`.
The separate clauses can therefore be merged into a single one:
```
  wk‚āÉ : (m n : ‚Ąē) ‚Üí suc m ‚Č§ n ‚Üí m ‚Č§ n
  wk‚āÉ  zero    zero          (base _)    = base refl
  wk‚āÉ  zero   (suc .zero)    (base refl) = step (base refl)
  wk‚āÉ (suc m)  zero          (base ())
  wk‚āÉ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wk‚āÉ m       (suc n)        (step m<n)  = step (wk‚āÉ m n m<n)
```
* As for the `base` clauses, the right-hand side of the second and the fourth clauses are the same, but the third clause seems to do something different.
* The third clause may be easier to handle because the domain is empty, so all definitions are extensionally equivalent; still, can we actually make it look the same with the other clauses?
* The first clause is also different, but has been mentioned above; this one seems trickier because `base refl` and `step (base refl)` look completely different.
* The transformations haven‚Äôt been finished, and we need more machinery.

The remaining clauses do not admit the same simplification. They are not recursive clauses, so there is no common recursive pattern to factor out. Instead, ... .

This suggests that the transformations from `wk'` to `wk` are more than reordering and merging clauses.
Those operations are enough to expose the uniformity of the `step` case, but they do not explain ... .
To see all the transformation steps, we need to make the underlying mechanism of pattern matching mechanism explicit.

### Basic analysis operators

Back to the transformations from `wk'` to `wk`, to see all the steps we need to make the underlying mechanism of dependent pattern matching explicit.
One way to make that mechanism explicit is to translate the clause-based definition into the corresponding eliminators, as in Goguen, McBride, and McKinna‚Äôs [*Eliminating dependent pattern matching*](https://doi.org/10.1007/11780274_27).
For example, the eliminator for `_‚Č§_` is:
```
  elim-‚Č§ : (P : (m n : ‚Ąē) ‚Üí m ‚Č§ n ‚Üí Set)
         ‚Üí ({m n : ‚Ąē} {m‚Č°n : m ‚Č° n} ‚Üí P m n (base m‚Č°n))
         ‚Üí ({m n : ‚Ąē} {m‚Č§n : m ‚Č§ n} ‚Üí P m n m‚Č§n ‚Üí P m (suc n) (step m‚Č§n))
         ‚Üí (m n : ‚Ąē) (m‚Č§n : m ‚Č§ n) ‚Üí P m n m‚Č§n
  elim-‚Č§ P pb ps m .m       (base refl) = pb
  elim-‚Č§ P pb ps m .(suc _) (step m‚Č§n)  = ps (elim-‚Č§ P pb ps m _ m‚Č§n)
```
But if we look at an actual program, in particular one in Epigram, we‚Äôll see that there‚Äôs some distance between the kind of program we want to write and eliminators.
For example, Section 2 of McBride‚Äôs [*Epigram: Practical programming with dependent types*](https://doi.org/10.1007/11546382_3) gave this standard example:
```
plus x y ‚áź rec x {
  plus x y ‚áź case x {
    plus  zero   y ‚áí y
    plus (suc x) y ‚áí suc (plus x y) } }
```
There are two things to observe here: (i) Conor chose to separate *recursion* and *case analysis*; (ii) the target to be recursed over or analysed (`x` in the example above) may be part of a larger input (`x` and `y` above).
But eliminators mix recursion and case analysis, and assume that the proof and its indices are the only things in the input ‚ÄĒ for example, `elim-‚Č§` ends with `(m n : ‚Ąē) (m‚Č§n : m ‚Č§ n) ‚Üí P m n m‚Č§n` instead of a general dependent function type `(őī : őĒ) ‚Üí T őī`.
So instead of translating pattern-matching programs directly to eliminators, Goguen et al derived separate recursion and case analysis operators from eliminators, and adapted them for translating general dependent functions.
(These operators could be thought of as giving a semantics to Epigram‚Äôs `rec` and `case`.)

We‚Äôll look at case analysis first.
The following `case-‚Č§` operator is the same as `elim-‚Č§` except that the induction hypothesis in the `step` method is dropped, so the `case-‚Č§` operator only performs a case analysis on a proof of `m ‚Č§ n`.
```
case-‚Č§ : (P : ‚ąÄ m n ‚Üí m ‚Č§ n ‚Üí Set)
       ‚Üí ((m n : ‚Ąē) (m‚Č°n : m ‚Č° n) ‚Üí P m n (base m‚Č°n))
       ‚Üí ((m n : ‚Ąē) (m‚Č§n : m ‚Č§ n) ‚Üí P m (suc n) (step m‚Č§n))
       ‚Üí (m n : ‚Ąē) (m‚Č§n : m ‚Č§ n) ‚Üí P m n m‚Č§n
case-‚Č§ P pb ps m n m‚Č§n = elim-‚Č§ P (őĽ {m‚āÄ n‚āÄ m‚āÄ‚Č°n‚āÄ} ‚Üí pb m‚āÄ n‚āÄ m‚āÄ‚Č°n‚āÄ) (őĽ {m‚āÄ n‚āÄ m‚āÄ‚Č§n‚āÄ} _ ‚Üí ps m‚āÄ n‚āÄ m‚āÄ‚Č§n‚āÄ) m n m‚Č§n
```
Like `elim-‚Č§`, the type of `case-‚Č§` ends with `(m n : ‚Ąē) (m‚Č§n : m ‚Č§ n) ‚Üí P m n m‚Č§n`, but we want to change that to a general dependent function type `(őī : őĒ) ‚Üí T őī`:
```
basic-case-‚Č§ : (őĒ : Set) (T : őĒ ‚Üí Set) (t : őĒ ‚Üí ő£[ m ‚ąą ‚Ąē ] ő£[ n ‚ąą ‚Ąē ] (m ‚Č§ n))
             ‚Üí ((őī : őĒ) (m n : ‚Ąē) (m‚Č°n : m ‚Č° n) ‚Üí (m , n , base m‚Č°n) ‚Č° t őī ‚Üí T őī)
             ‚Üí ((őī : őĒ) (m n : ‚Ąē) (m‚Č§n : m ‚Č§ n) ‚Üí (m , suc n , step m‚Č§n) ‚Č° t őī ‚Üí T őī)
             ‚Üí (őī : őĒ) ‚Üí T őī
basic-case-‚Č§ őĒ T t m‚āĀ m‚āā őī = let (m , n , m‚Č§n) = t őī in case-‚Č§ (őĽ m n m‚Č§n ‚Üí (őī : őĒ) ‚Üí (m , n , m‚Č§n) ‚Č° t őī ‚Üí T őī) (őĽ m n m‚Č°n őī ‚Üí m‚āĀ őī m n m‚Č°n) (őĽ m n m‚Č§n őī ‚Üí m‚āā őī m n m‚Č§n) m n m‚Č§n őī refl
```
Goguen et al called such an operator ‚Äėbasic analysis‚Äô, hence the name `basic-case-‚Č§`.
The first two arguments of `basic-case-‚Č§` specify what `őĒ` and `T` are.
The next argument `t` extracts the proof and its indices from an input `őī : őĒ` to be analysed.
In the two methods, we add equality constraints stating that the extracted proof and its indices are specialised to `base` and `step` between the input `(őī : őĒ)` and the output `T őī`.
So when we use `basic-case-‚Č§` to write a function `(őī : őĒ) ‚Üí T őī` that analyses `t őī`, we split the programming problem into two cases; in each case, we still write a function from `őī : őĒ` to `T őī` but can assume an additional equality constraint that `t őī` has a more specialised form.

* How `basic-case-‚Č§` is proved from `case-‚Č§`: The motive is then rebuilt so that the result is still stated as `T őī`, while each branch carries the equality constraint connecting the local constructor form back to the extracted data.
* Goguen et al gave a general construction for deriving basic analysis operators from ‚Äėelimination operators‚Äô (which hopefully we don‚Äôt have to explain in detail)
* (written below) Introduce `rec-‚Č§` and apply the general construction to `rec-‚Č§` to get `basic-rec-‚Č§`

Now, we move on to recursion.
For case analysis, the methods only need to handle the immediate constructor shape of the proof being analysed.
For recursion, however, the methods also need access to the results of recursive calls on structurally smaller proofs.

* results on *all* structurally smaller proofs ‚ÄĒ strong induction
* We therefore first define a type describing what recursive information is available below ‚ÄĒstructurally smaller than‚ÄĒ a given proof.

Defined below, `Below-‚Č§ P m n m‚Č§n` is an iterated tuple type asserting that the predicate `P` holds for every value structurally smaller than the current proof `m‚Č§n`:
```
Below-‚Č§ : (P : ‚ąÄ m n ‚Üí m ‚Č§ n ‚Üí Set) ‚Üí ‚ąÄ m n ‚Üí m ‚Č§ n ‚Üí Set
Below-‚Č§ P m .m       (base refl) = ‚ä§
Below-‚Č§ P m .(suc _) (step m‚Č§n)  = Below-‚Č§ P m _ m‚Č§n √ó P m _ m‚Č§n
```
In the `base` case, there is no smaller proof, so the tuple is trivial.
In the `step` case, the proof contains a predecessor proof `m‚Č§n`, so the tuple contains both the recursive information below that predecessor and the result of `P` for the predecessor itself.
Then we can define the type of the recursion operator, where we use `Below-‚Č§` as the type of the induction hypotheses:
```
rec-‚Č§ : (P : ‚ąÄ m n ‚Üí m ‚Č§ n ‚Üí Set)
      ‚Üí (‚ąÄ m n ‚Üí (m‚Č§n : m ‚Č§ n) ‚Üí Below-‚Č§ P m n m‚Č§n ‚Üí P m n m‚Č§n)
      ‚Üí  ‚ąÄ m n ‚Üí (m‚Č§n : m ‚Č§ n) ‚Üí P m n m‚Č§n
```
The operational intuition of strong induction is to never forget previous results and accumulate them in a tuple instead, which is used as the strengthened induction hypotheses.
A clean way of implementing `rec-‚Č§` is to write a function constructing such tuples first:
```
below-‚Č§ : (P : ‚ąÄ m n ‚Üí m ‚Č§ n ‚Üí Set)
        ‚Üí (p : ‚ąÄ m n ‚Üí (m‚Č§n : m ‚Č§ n) ‚Üí Below-‚Č§ P m n m‚Č§n ‚Üí P m n m‚Č§n)
        ‚Üí ‚ąÄ m n ‚Üí (m‚Č§n : m ‚Č§ n) ‚Üí Below-‚Č§ P m n m‚Č§n
below-‚Č§ P p m .m       (base refl) = tt
below-‚Č§ P p m .(suc _) (step m‚Č§n)  = let b = below-‚Č§ P p m _ m‚Č§n
                                      in (b , p m _ m‚Č§n b)
```
The type of `below-‚Č§` is the same as that of `rec-‚Č§` except that the result type is the `Below` tuple type.
In the `base` case, there are no smaller proofs, so the tuple is just `tt`.
In the `step` case, we first build the tuple `b` below `m‚Č§n` recursively (which are the strengthened hypotheses), and then extend the tuple with the result for `m‚Č§n` itself, obtained by applying the method `p` to `b` (the strengthened induction hypotheses).
Then all it takes to get from `below-‚Č§` to `rec-‚Č§` is a final application of `p`:
```
rec-‚Č§ : (P : ‚ąÄ m n ‚Üí m ‚Č§ n ‚Üí Set)
      ‚Üí (‚ąÄ m n ‚Üí (m‚Č§n : m ‚Č§ n) ‚Üí Below-‚Č§ P m n m‚Č§n ‚Üí P m n m‚Č§n)
      ‚Üí  ‚ąÄ m n ‚Üí (m‚Č§n : m ‚Č§ n) ‚Üí P m n m‚Č§n
rec-‚Č§ P p m n m‚Č§n = p m n m‚Č§n (below-‚Č§ P p m n m‚Č§n)
```

Same as `case-‚Č§`, we want `rec-‚Č§` to end with a general dependent function type `(őī : őĒ) ‚Üí T őī`, so we apply Goguen et al‚Äôs construction to `rec-‚Č§` and get
```
basic-rec-‚Č§ : (őĒ : Set) (T : őĒ ‚Üí Set) (t : (őī : őĒ) ‚Üí ő£[ m ‚ąą ‚Ąē ] ő£[ n ‚ąą ‚Ąē ] (m ‚Č§ n))
            ‚Üí ((őī : őĒ) (m n : ‚Ąē) (m‚Č§n : m ‚Č§ n) ‚Üí (b : Below-‚Č§ (őĽ m' n' m‚Č§n' ‚Üí (őī : őĒ) ‚Üí (m' , n' , m‚Č§n') ‚Č° t őī ‚Üí T őī) m n m‚Č§n) ‚Üí (m , n , m‚Č§n) ‚Č° t őī ‚Üí T őī)
            ‚Üí  (őī : őĒ) ‚Üí T őī
basic-rec-‚Č§ őĒ T t p őī = let (m , n , m‚Č§n) = t őī in rec-‚Č§ (őĽ m' n' m‚Č§n' ‚Üí (őī : őĒ) ‚Üí (m' , n' , m‚Č§n') ‚Č° t őī ‚Üí T őī) (őĽ m n m‚Č§n b őī ‚Üí p őī m n m‚Č§n b) m n m‚Č§n őī refl
```
When we use `basic-rec-‚Č§` to write a function `(őī : őĒ) ‚Üí T őī` that recurses on `t őī`, first we specify `őĒ`, `T`, and `t`, and then we write a function with the same input and output but extended with additional arguments.
The first three additional arguments `m`, `n`, and `m‚Č§n` can be thought of as the results of pattern-matching `t őī` with a tuple because there is an equality constraint `(m , n , m‚Č§n) ‚Č° t őī` near the end.
Then we get a `Below-‚Č§` argument that allows us to make recursive calls as long as the `m‚Č§n` proof gets structurally smaller.
Such a recursive call can change the input `őī` as needed (to something larger or even unrelated), but the `m‚Č§n` part of `őī` should get smaller for the recursive call to be legitimate.
So the `Below-‚Č§` argument contains functions of type `(őī : őĒ) ‚Üí (m' , n' , m‚Č§n') ‚Č° t őī ‚Üí T őī` for all `(m' , n' , m‚Č§n')` that are smaller than `(m , n , m‚Č§n)`.

In the next section we will see concrete examples of translating pattern-matching programs to basic analysis, namely `wk` and `wk‚āĀ` (which is close to `wk'`).
Before moving forward to the translation, we need to apply the above techniques to another datatype `‚ĄēPair = ‚Ąē √ó ‚Ąē`, since `wk‚āĀ`‚Äôs first pattern-matching target is its pair of natural-number arguments.
The eliminator we start with matches both natural numbers with `zero` or `suc`, and allows recursion whenever one or both numbers get structurally smaller:
```
elim-‚ĄēPair : (P : ‚Ąē √ó ‚Ąē ‚Üí Set)
           ‚Üí P (zero , zero)
           ‚Üí ({n : ‚Ąē} ‚Üí P (zero , n) ‚Üí P (zero , suc n))
           ‚Üí ({m : ‚Ąē} ‚Üí P (m , zero) ‚Üí P (suc m , zero))
           ‚Üí ({m n : ‚Ąē} ‚Üí P (m , n) ‚Üí P (suc m , n) ‚Üí P (m , suc n) ‚Üí P (suc m , suc n))
           ‚Üí ((m , n) : ‚Ąē √ó ‚Ąē) ‚Üí P (m , n)
elim-‚ĄēPair p pz ps‚āô ps‚āė ps (zero  , zero)  = pz
elim-‚ĄēPair p pz ps‚āô ps‚āė ps (zero  , suc n) = ps‚āô (elim-‚ĄēPair p pz ps‚āô ps‚āė ps (zero , n))
elim-‚ĄēPair p pz ps‚āô ps‚āė ps (suc m , zero ) = ps‚āė (elim-‚ĄēPair p pz ps‚āô ps‚āė ps (m , zero))
elim-‚ĄēPair p pz ps‚āô ps‚āė ps (suc m , suc n) = ps (elim-‚ĄēPair p pz ps‚āô ps‚āė ps (m , n)) (elim-‚ĄēPair p pz ps‚āô ps‚āė ps (suc m , n)) (elim-‚ĄēPair p pz ps‚āô ps‚āė ps (m , suc n))
```
In the last case, it allows recursive access to the three smaller pairs: `(m , n)`, `(suc m , n)`, and `(m , suc n)`.
Next, we split `elim-‚ĄēPair` into two operators, `case-‚ĄēPair` for case analysis,
```
case-‚ĄēPair : (P : ‚Ąē √ó ‚Ąē ‚Üí Set)
           ‚Üí P (zero , zero)
           ‚Üí ((n : ‚Ąē) ‚Üí P (zero , suc n))
           ‚Üí ((m : ‚Ąē) ‚Üí P (suc m , zero))
           ‚Üí ((m n : ‚Ąē) ‚Üí P (suc m , suc n))
           ‚Üí (m n : ‚Ąē) ‚Üí P (m , n)
 case-‚ĄēPair P pz pn pm ps zero zero = pz
 case-‚ĄēPair P pz pn pm ps zero (suc n) = pn n
 case-‚ĄēPair P pz pn pm ps (suc m) zero = pm m
 case-‚ĄēPair P pz pn pm ps (suc m) (suc n) = ps m n
```
and `rec-‚ĄēPair` for recursive call, following the same recipe as above:
```
Below-‚ĄēPair : (P : ‚Ąē √ó ‚Ąē ‚Üí Set) ‚Üí ‚Ąē ‚Üí ‚Ąē ‚Üí Set
Below-‚ĄēPair P  zero    zero   = ‚ä§
Below-‚ĄēPair P  zero   (suc n) = Below-‚ĄēPair P zero n √ó P (zero , n)
Below-‚ĄēPair P (suc m)  zero   = Below-‚ĄēPair P m zero √ó P (m , zero)
Below-‚ĄēPair P (suc m) (suc n) = (Below-‚ĄēPair P m n √ó P (m , n)) √ó (Below-‚ĄēPair P (suc m) n √ó P (suc m , n)) √ó (Below-‚ĄēPair P m (suc n) √ó P (m , suc n))

below-‚ĄēPair : (P : ‚Ąē √ó ‚Ąē ‚Üí Set)
            ‚Üí (p : (m n : ‚Ąē) ‚Üí Below-‚ĄēPair P m n ‚Üí P (m , n))
            ‚Üí (m n : ‚Ąē) ‚Üí Below-‚ĄēPair P m n
below-‚ĄēPair P p  zero    zero   = tt
below-‚ĄēPair P p  zero   (suc n) = let b = below-‚ĄēPair P p zero n in (b , p zero n b)
below-‚ĄēPair P p (suc m)  zero   = let b = below-‚ĄēPair P p m zero in (b , p m zero b)
below-‚ĄēPair P p (suc m) (suc n) = let b = below-‚ĄēPair P p m n
                                      b‚āô = below-‚ĄēPair P p (suc m) n
                                      b‚āė = below-‚ĄēPair P p m (suc n)
                                  in ((b , p m n b) , (b‚āô , p (suc m) n b‚āô) , (b‚āė , p m (suc n) b‚āė))

* In general, the purpose of the basic operators is to generalise the ordinary case and recursion operators from the data being analysed to an arbitrary input context.
A basic case operator introduces the arguments required by each constructor case, together with an equality constraint relating that local constructor form to the data extracted from the global input.
A basic recursion operator introduces the recursive information available below the targeted data, again with an equality constraint relating the local recursive problem to the extracted data.

### Basic analysis translation

We now illustrate the translation using `wk`:
```

The function takes three inputs: `m : ‚Ąē`, `n : ‚Ąē`, and `m<n : suc m ‚Č§ n`.
In the Agda definition, case analysis is more visible, but the first step of the translation is to invoke a basic recursion operator.
In this case, the target on which the function recurses is `m<n`, so we use `basic-rec-‚Č§`:

In the translation, we package the three inputs into a triple `(m , n , m<n) : ő£[ m ‚ąą ‚Ąē ] ő£[ n ‚ąą ‚Ąē ] suc m ‚Č§ n` (whose type is given as the first argument to `basic-rec-‚Č§`) and pass the triple into the operator as the last argument.
The output type of the function is specified by the second argument `őĽ (m , n , _  ) ‚Üí m ‚Č§ n`.
The third argument specifies the target on which to perform recursion, which belongs to the indexed datatype `_‚Č§_`, so the extraction function returns the `_‚Č§_` data along with its surrounding indices: `(suc m , n , m<n)`.
What remains is the method; its inputs, after some simplification, are the original three plus a `Below` argument `b`.
In general (for a recursive function), we apply basic recursion operators as the outermost layers and pass the `Below` arguments to subsequent basic case-analysis operators, which perform the actual case split.

---

* (written below) A general pattern: basic-rec then basic-case, so that the Below argument is a part of input to basic-case, and can be expanded by the case analysis

Once we have the basic case analysis and recursion operators, we can translate an ordinary pattern-matching definition into a program constructed using these operators.
The translation begins by choosing the first targeted data, namely the data on which the original definition performs its first pattern match.
The targeted data is then updated, and further basic case-analysis operators are applied until every branch can be solved.
Notice that recursion is introduced before case analysis.
This makes the recursive information available first, so that the `Below` argument can be passed into the subsequent case analysis and expanded according to the specialised shape of the targeted data.
As a result, the recursive results required in each branch are already available.

### The program transformations from `wk'` to `wk` explained

We now return to the unresolved question in the transformation from `wk'` to `wk`: why can the `step` branches be made uniform simply by reordering and merging clauses, whereas the `base` branches cannot?
Back in the clause-based version, the transformation gets stuck when we try to eliminate redundant case analysis. This elimination succeeds for the `step` branches, but not for the `base` branches.
Moreover, the main difference between the two pattern matching strategies (`wk` and `wk'`) is the case analysis order of the data (`(m , n)` and `suc m ‚Č§ n`).
Using the technique introduced above, we now have more flexibility by manipulating the basic analysis operators, and hence can formulate the following two transformation laws,
```
   lhs ‚áí rhs
=  lhs ‚áź case ... {
     lhs'  ‚áí rhs'
     lhs'' ‚áí rhs'' }
```
for redundant case analysis, and
```
   case x {
      x0 ‚áí case y { y0 ‚áí ...
                    y1 ‚áí ... }
      x1 ‚áí case y { y0 ‚áí ...
                    y1 ‚áí ... } }
 = case y {
      y0 ‚áí case x { x0 ‚áí ...
                    x1 ‚áí ... }
      y1 ‚áí case x { x0 ‚áí ...
                    x1 ‚áí ... } }
```
for swapping nested case analysis.
These two laws correspond to the two observations above.
The redundant case-analysis law lets us revisit the failed elimination (on the `base` branches) in the clause-based transformation, while the case-swapping law captures the difference in analysis order between `wk` and `wk‚Ä≤`.
Thus, the transformation can be discussed in terms of these two operator-level laws rather than only in terms of clause rearrangement.

Back to the question at the beginning of this section, we can now explain the situation more precisely:
1. The redundant-case elimination in the `step` branches succeeds only accidentally in the clause-based version. Since Agda automatically omits the impossible branches, we only see the two surviving `step` clauses, and these can be merged after reordering. In the basic-analysis version, the omitted branches reappear as contradiction branches, making clear that the clause-based transformation was relying on Agda‚Äôs omission of impossible cases.
2. The basic-analysis translation lets us split case analysis and unification, which are combined into a single step in ordinary pattern matching. This is the crucial technique for completing the transformation: by delaying the equality constraints, we can control where unification takes place. The unification strategy decides how the right-hand side of the program can be written, and therefore can perform different definitions of the same function.
3. Apart from unification, the two basic-analysis transformation laws stated above correspond to case-reordering and redundant-case elimination in clause-based transformation, which determine only the outer structure of the program transformation but not how the branch bodies are constructed.

One limitation should be noted here.
These laws hold at the level of basic analysis operators, not directly at the surface pattern matching level.
This is especially important for the delayed-constraint transformation: ordinary pattern matching does not expose equality constraints separately from unification, so this step cannot be stated cleanly at the clause level.

### A more general story

To put the post into context:
In Shin‚Äôs [JFP pearl](https://doi.org/10.1017/S0956796824000145) on top-down vs bottom-up sublist computation, he gave an equational specification (3.1) and a derivation of a function which he called `upgrade`; subsequently I expressed Shin‚Äôs specification (3.1) in a dependent type and reimplemented (and simultaneously verified) the function, which I called `retabulate` in the [short story paper](https://doi.org/10.48550/arXiv.2503.04001).
Shu-Ting and I were intrigued by how different the two solutions look.
After Shu-Ting‚Äôs numerous experiments, gradually we suspected that the first solution can be systematically transformed into the second, and a key step is switching from a two-level analysis on indices and then indexed data to a direct analysis on indexed data, as illustrated by the `wk` example.

The original example looks more like the following (which is still a simplified version; the complete code is included in the attached Agda file):
```
up : ‚ąÄ k ‚Üí BTree A n (suc k) ‚Üí 2 ‚Č§ n ‚Üí suc (suc k) ‚Č§ n ‚Üí BTree (Vec A (suc (suc k))) n (suc (suc k))
up k (tipN x)                        l‚āĀ          l‚āā         = ‚ä•-elim (n‚Čģn (suc k) (‚Č§‚Ä≤‚áí‚Č§ l‚āā))
up k (node (tip0 x)    (tipN y))     l‚āĀ          l‚āā         = tipN (x ‚ą∑ y ‚ą∑ [])
up k (node (tip0 x)    (node u u'))  l‚āĀ          l‚āā         = node (mapB (őĽ q ‚Üí x ‚ą∑ q ‚ą∑ []) (node u u')) (up zero (node u u') (s‚Č§‚Ä≤s (bounded u')) (s‚Č§‚Ä≤s (bounded u')))
up k (node (tipN x)     u)           l‚āĀ          l‚āā         = ‚ä•-elim (n‚Čģn (suc k) (‚Č§‚Ä≤‚áí‚Č§ l‚āā))
up k (node (node t t') (tipN x))    (step l‚āĀ)    l‚āā         = tipN (unTip (up _ (node t t') l‚āĀ (base refl)) ‚ą∑ ≥ x)
up k (node (node t t') (node u u')) (base refl)  l‚āā         = ‚ä•-elim (n‚Čģ0 (‚Č§-pred (‚Č§-pred (‚Č§‚Ä≤‚áí‚Č§ l‚āā))))
up k (node (node t t') (node u u')) (step l‚āĀ)   (base refl) = ‚ä•-elim (unbounded u')
up k (node (node t t') (node u u')) (step l‚āĀ)   (step l‚āā)   = node (zipBW _‚ą∑ ≥_ (up _ (node t t') l‚āĀ (‚Č§‚áí‚Č§‚Ä≤ (<‚áí‚Č§ (‚Č§‚Ä≤‚áí‚Č§ l‚āā)))) (node u u')) (up _ (node u u') l‚āĀ l‚āā)
```
This `up` function is a dependently typed reimplementation of Shin‚Äôs `upgrade` function.
Roughly speaking, `up` transforms a tree to another by performing pattern matching on the input tree; the trees have indexed types that express shape constraints, and on the indices there are some more constraints, which are conditions assumed by Shin‚Äôs specification (3.1).
Upon closer inspection, Shin‚Äôs derivation starts with analyses on the indices and the constraints/conditions.
If `up` used the same case analysis strategy, with the arguments rearranged in the order in which they are analysed, it would look like
```
up' : 2 ‚Č§ n ‚Üí ‚ąÄ k ‚Üí suc (suc k) ‚Č§ n ‚Üí BTree A n (suc k) ‚Üí BTree (Vec A (suc (suc k))) n (suc (suc k))
up' (base refl) k      (base refl) (node (tip0 x)    (tipN y))    = tipN (x ‚ą∑ y ‚ą∑ [])
up' (base refl) k      (step l‚āā)    t                             with step (base ()) ‚Üź l‚āā
up' (step l‚āĀ)   k      (base refl) (node (tip0 x)    (tipN y))    with step (base ()) ‚Üź l‚āĀ
up' (step l‚āĀ) .(suc _) (base refl) (node (node t t') (tipN x))    = tipN (unTip (up' l‚āĀ _ (base refl) (node t t')) ‚ą∑ ≥ x)
up' (step l‚āĀ) .(suc _) (base refl) (node (node t t') (node u u')) = ‚ä•-elim (unbounded u')
up' (step l‚āĀ)   zero   (step l‚āā)   (tipN x)                       = ‚ä•-elim (n‚Čģ0 (‚Č§‚Ä≤‚áí‚Č§ l‚āā))
up' (step l‚āĀ)   zero   (step l‚āā)   (node (tip0 x)     u)          = node (mapB (őĽ q ‚Üí x ‚ą∑ q ‚ą∑ []) u) (up' l‚āĀ zero l‚āā u)
up' (step l‚āĀ)  (suc k) (step l‚āā)   (tipN x)                       = ‚ä•-elim (n‚Čģn (suc (suc k)) (‚Č§‚Ä≤‚áí‚Č§ (step l‚āā)))
up' (step l‚āĀ)  (suc k) (step l‚āā)   (node t            u)          = node (zipBW _‚ą∑ ≥_ (up' l‚āĀ k (‚Č§‚áí‚Č§‚Ä≤ (‚Č§-pred (‚Č§‚Ä≤‚áí‚Č§ (step l‚āā)))) t) u) (up' l‚āĀ (suc k) l‚āā u)
```
We believe that `up` and `up'` are related by the simple transformations described in this post.
A more complete story involves more transformations:
Shin started with an equational specification, which, as a type, looks like
```
2 ‚Č§ n ‚Üí ‚ąÄ k ‚Üí suc (suc k) ‚Č§ n ‚Üí ‚ąÄ t ‚Üí ch k xs ‚Č° t ‚Üí ‚čĮ
```
The involved functions can be replaced by inductively defined relations, for example
```
2 ‚Č§ n ‚Üí ‚ąÄ k ‚Üí suc (suc k) ‚Č§ n ‚Üí ‚ąÄ t ‚Üí Ch k xs t ‚Üí ‚čĮ
```
where the equation involving `ch` is replaced by a relation `Ch` whose inhabitants are [execution traces](/blog/0043/) of `ch`.
The output tree `t` is a simplified representation of those traces, so `Ch` looks like an algebraic ornamentation of `BTree`, which brings us closer to `up'`.
Our hunch is that by applying a somewhat long series of transformations we may be able to get from Shin‚Äôs equational proof to the `retabulate` function on trees; each of the transformations may not be surprising, but the overall effect would be.
Alas, this means that if we want to write a paper, we‚Äôll need to cover quite a few transformations to make the paper interesting enough, but the workload seems too much.
It might be worth a try if we could find more convincing examples, but examples also turned out to be difficult to conceive.
So this is where we have to stop, write things up, and hope that suitable examples and follow-up ideas will show up some day.

On the other hand, this was a nice opportunity for us to start studying the underlying mechanisms that make dependently typed programming practical (compared with, for example, programming directly in Martin-L√∂f type theory).
We read Goguen et al‚Äôs [*Eliminating dependent pattern matching*](https://doi.org/10.1007/11780274_27) and followed the reference to Conor‚Äôs [*Elimination with a motive*](https://doi.org/10.1007/3-540-45842-5_13), which was probably where the basic analysis idea was first published (although we actually like Goguen et al‚Äôs succinct presentation more).
In these translations and our final version of transformations, Fording has been shown to be a technique that‚Äôs more important than it looks, but with the advent of homotopy type theory it‚Äôs less clear how harmless Fording is.
Even though I‚Äôm not eager to embrace the full generality of homotopy type theory, it‚Äôs still important to be aware of the big picture, so at some point I‚Äôll need to read the sequel [*Eliminating dependent pattern matching without K*](https://doi.org/10.1017/S0956796816000174) as well as [*Proof-relevant unification: Dependent pattern matching with only the axioms of your type theory*](https://doi.org/10.1017/S095679681800014X) by Cockx et al.
Also written by Cockx et al (who have revised all the papers for JFP ‚ÄĒ very impressive) is [*Elaborating dependent (co)pattern matching: No pattern left behind*](https://doi.org/10.1017/S0956796819000182), but I‚Äôm less sure about this one about (Agda‚Äôs) case trees because I‚Äôve found Epigram‚Äôs approach to be clearer (without having to rely on argument reordering to try to express case analysis strategies) and more general/extensible, as explained in [*The view from the left*](https://doi.org/10.1017/S0956796803004829).
