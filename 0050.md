## Dependently typed program transformations

(*This post is jointly written with Shu-Ting Hsu.*)

While writing Agda programs, pattern matching plays an important role in shaping the structure of a definition вАФ that is, applying different pattern matching strategies to the same function leads to different program structures corresponding to different ways of performing case analyses.
In particular, when working with an indexed data type, a beginner may analyse the indices first but end up with a more complex program, and then they are usually taught that it can be more straightforward to analyse the data.
For example, consider this definition of `_вЙ§_` (which is a renamed version of `_вЙ§вА≤_` from the standard library)
```
data _вЙ§_ : вДХ вЖТ вДХ вЖТ Set where
  base : m вЙ° n вЖТ m вЙ§ n
  step : m вЙ§ n вЖТ m вЙ§ suc n
```
and suppose that we want to weaken `suc m вЙ§ n` (that is, `m < n`) to `m вЙ§ n`:
```
wk : (m n : вДХ) вЖТ suc m вЙ§ n вЖТ m вЙ§ n
wk m .(suc m) (base refl)    = step (base refl)
wk m .(suc n) (step {n} m<n) = step (wk m n m<n)
```
This definition performs a case analysis on the proof `suc m вЙ§ n`.
To emphasise that the pattern matching on the second argument is not performed by the programmer but forced by unification, we use dot patterns explicitly (and therefore need to explicitly bind `n` in the `step` constructor in the second clause).
A beginner might be unfamiliar with the idea of induction on proofs though, and might write the following version that analyses the indices `m` and `n` first:
```
wk' : (m n : вДХ) вЖТ suc m вЙ§ n вЖТ m вЙ§ n
wk'  zero    zero          m<n         = base refl
wk'  zero   (suc .zero)    (base refl) = step (base refl)
wk'  zero   (suc n)        (step m<n)  = step (wk' zero n m<n)
wk' (suc m)  zero          (base ())
wk' (suc m) (suc .(suc m)) (base refl) = step (base refl)
wk' (suc m) (suc n)        (step m<n)  = step (wk' (suc m) n m<n)
```
Here, `wk'` is finished with six cases, and is more complex than `wk`.
In all the cases except the first one,

* once the indices `m` and `n` have been fixed, the shape of `suc m вЙ§ n` is already forced. (still need to do the matching explicitly)
  * the surrounding index patterns already determine which proof forms are possible
* `base refl` vs `step (base refl)` in the first case

Thus `wk` and `wk'` have different case analysis structures, even though extensionally they define the same function.
So, what is the connection between them? By transforming `wk'` step by step into `wk`, we can make those structural differences explicit.

### Transforming `wk'` to `wk`

The first step is to fully expand the two levels of case analyses in every clause.
In the case of `wk'`, only the first clause still needs to be expanded:
```
  wkвВБ : (m n : вДХ) вЖТ suc m вЙ§ n вЖТ m вЙ§ n
  wkвВБ  zero    zero          (base _)    = base refl
  wkвВБ  zero   (suc .zero)    (base refl) = step (base refl)
  wkвВБ  zero   (suc n)        (step m<n)  = step (wkвВБ zero n m<n)
  wkвВБ (suc m)  zero          (base ())
  wkвВБ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wkвВБ (suc m) (suc n)        (step m<n)  = step (wkвВБ (suc m) n m<n)
```
Next, we reorder the clauses so that the clauses with the same patterns for  `suc m вЙ§ n` are grouped together, to indicate that the case analysis on the indexed data `suc m вЙ§ n` is now performed first (although the case tree generated by Agda wonвАЩt reflect that):
```
  wkвВВ : (m n : вДХ) вЖТ suc m вЙ§ n вЖТ m вЙ§ n
  wkвВВ  zero    zero          (base _)    = base refl
  wkвВВ  zero   (suc .zero)    (base refl) = step (base refl)
  wkвВВ (suc m)  zero          (base ())
  wkвВВ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wkвВВ  zero   (suc n)        (step m<n)  = step (wkвВВ  zero   n m<n)
  wkвВВ (suc m) (suc n)        (step m<n)  = step (wkвВВ (suc m) n m<n)
```
In the `step` clauses, the right-hand sides do not actually depend on whether the first argument is `zero` or `suc m` вАФ the difference is only in the first arguments of the recursive calls, but what they both do is copying the first argument, regardless of whether thatвАЩs `zero` or `suc m`.
The separate clauses can therefore be merged into a single one:
```
  wkвВГ : (m n : вДХ) вЖТ suc m вЙ§ n вЖТ m вЙ§ n
  wkвВГ  zero    zero          (base _)    = base refl
  wkвВГ  zero   (suc .zero)    (base refl) = step (base refl)
  wkвВГ (suc m)  zero          (base ())
  wkвВГ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wkвВГ m       (suc n)        (step m<n)  = step (wkвВГ m n m<n)
```
* As for the `base` clauses, the right-hand side of the second and the fourth clauses are the same, but the third clause seems to do something different.
* The third clause may be easier to handle because the domain is empty, so all definitions are extensionally equivalent; still, can we actually make it look the same with the other clauses?
* The first clause is also different, but has been mentioned above; this one seems trickier because `base refl` and `step (base refl)` look completely different.
* The transformations havenвАЩt been finished, and we need more machinery.

The remaining clauses do not admit the same simplification. They are not recursive clauses, so there is no common recursive pattern to factor out. Instead, ... .

### Basic analysis operators

Back to the transformations from `wk'` to `wk`, to see all the steps we need to make the underlying mechanism of dependent pattern matching explicit.
One way to make that mechanism explicit is to translate the clause-based definition into the corresponding eliminators, as in Goguen, McBride, and McKinnaвАЩs [*Eliminating dependent pattern matching*](https://doi.org/10.1007/11780274_27).
For example, the eliminator for `_вЙ§_` is:
```
  elim-вЙ§ : (P : (m n : вДХ) вЖТ m вЙ§ n вЖТ Set)
         вЖТ ({m n : вДХ} {mвЙ°n : m вЙ° n} вЖТ P m n (base mвЙ°n))
         вЖТ ({m n : вДХ} {mвЙ§n : m вЙ§ n} вЖТ P m n mвЙ§n вЖТ P m (suc n) (step mвЙ§n))
         вЖТ (m n : вДХ) (mвЙ§n : m вЙ§ n) вЖТ P m n mвЙ§n
  elim-вЙ§ P pb ps m .m       (base refl) = pb
  elim-вЙ§ P pb ps m .(suc _) (step mвЙ§n)  = ps (elim-вЙ§ P pb ps m _ mвЙ§n)
```
But if we look at an actual program, in particular one in Epigram, weвАЩll see that thereвАЩs some distance between the kind of program we want to write and eliminators.
For example, Section 2 of McBrideвАЩs [*Epigram: Practical programming with dependent types*](https://doi.org/10.1007/11546382_3) gave this standard example:
```
plus x y вЗР rec x {
  plus x y вЗР case x {
    plus  zero   y вЗТ y
    plus (suc x) y вЗТ suc (plus x y) } }
```
There are two things to observe here: (i) Conor chose to separate *recursion* and *case analysis*; (ii) the target to be recursed over or analysed (`x` in the example above) may be part of a larger input (`x` and `y` above).
But eliminators mix recursion and case analysis, and assume that the proof and its indices are the only things in the input вАФ for example, `elim-вЙ§` ends with `(m n : вДХ) (mвЙ§n : m вЙ§ n) вЖТ P m n mвЙ§n` instead of a general dependent function type `(ќі : ќФ) вЖТ T ќі`.
So instead of translating pattern-matching programs directly to eliminators, Goguen et al derived separate recursion and case analysis operators from eliminators, and adapted them for writing general dependent functions.
(These operators could be thought of as giving a semantics to EpigramвАЩs `rec` and `case`.)

WeвАЩll look at case analysis first.
The following `case-вЙ§` operator is the same as `elim-вЙ§` except that the induction hypothesis in the `step` method is dropped, so the `case-вЙ§` operator only performs a case analysis on a proof of `m вЙ§ n`.
```
case-вЙ§ : (P : вИА m n вЖТ m вЙ§ n вЖТ Set)
       вЖТ ((m n : вДХ) (mвЙ°n : m вЙ° n) вЖТ P m n (base mвЙ°n))
       вЖТ ((m n : вДХ) (mвЙ§n : m вЙ§ n) вЖТ P m (suc n) (step mвЙ§n))
       вЖТ (m n : вДХ) (mвЙ§n : m вЙ§ n) вЖТ P m n mвЙ§n
case-вЙ§ P pb ps m n mвЙ§n = elim-вЙ§ P (ќї {mвВА nвВА mвВАвЙ°nвВА} вЖТ pb mвВА nвВА mвВАвЙ°nвВА) (ќї {mвВА nвВА mвВАвЙ§nвВА} _ вЖТ ps mвВА nвВА mвВАвЙ§nвВА) m n mвЙ§n
```
Like `elim-вЙ§`, the type of `case-вЙ§` ends with `(m n : вДХ) (mвЙ§n : m вЙ§ n) вЖТ P m n mвЙ§n`, but we want to change that to a general dependent function type `(ќі : ќФ) вЖТ T ќі`:
```
basic-case-вЙ§ : (ќФ : Set) (T : ќФ вЖТ Set) (t : ќФ вЖТ ќ£[ m вИИ вДХ ] ќ£[ n вИИ вДХ ] (m вЙ§ n))
             вЖТ ((ќі : ќФ) (m n : вДХ) (mвЙ°n : m вЙ° n) вЖТ (m , n , base mвЙ°n) вЙ° t ќі вЖТ T ќі)
             вЖТ ((ќі : ќФ) (m n : вДХ) (mвЙ§n : m вЙ§ n) вЖТ (m , suc n , step mвЙ§n) вЙ° t ќі вЖТ T ќі)
             вЖТ (ќі : ќФ) вЖТ T ќі

```
Goguen et al called such an operator вАШbasic analysisвАЩ, hence the name `basic-case-вЙ§`.
The first two arguments of `basic-case-вЙ§` specify what `ќФ` and `T` are.
The next argument `t` extracts the proof and its indices from an input `ќі : ќФ` to be analysed.
In the two methods, we add equality constraints stating that the extracted proof and its indices are specialised to `base` and `step` between the input `(ќі : ќФ)` and the output `T ќі`.
So when we use `basic-case-вЙ§` to write a function `(ќі : ќФ) вЖТ T ќі` that analyses `t ќі`, we split the programming problem into two cases; in each case, we still write a function from `ќі : ќФ` to `T ќі` but can assume an additional equality constraint that `t ќі` has a more specialised form.

* How `basic-case-вЙ§` is proved from `case-вЙ§`: The motive is then rebuilt so that the result is still stated as `T ќі`, while each branch carries the equality constraint connecting the local constructor form back to the extracted data.

Now, we move on to recursion.
For case analysis, the methods only need to handle the immediate constructor shape of the proof being analysed.
For recursion, however, the methods also need access to the results of recursive calls on structurally smaller proofs.

* results on *all* structurally smaller proofs вАФ strong induction
* We therefore first define a type describing what recursive information is available below вАФstructurally smaller thanвАФ a given proof.

Defined below, `Below-вЙ§ P m n mвЙ§n` is an iterated tuple type asserting that the predicate `P` holds for every value structurally smaller than the current proof `mвЙ§n`:
```
Below-вЙ§ : (P : вИА m n вЖТ m вЙ§ n вЖТ Set) вЖТ вИА m n вЖТ m вЙ§ n вЖТ Set
Below-вЙ§ P m .m       (base refl) = вК§
Below-вЙ§ P m .(suc _) (step mвЙ§n)  = Below-вЙ§ P m _ mвЙ§n √Ч P m _ mвЙ§n
```
In the `base` case, there is no smaller proof, so the tuple is trivial.
In the `step` case, the proof contains a predecessor proof `mвЙ§n`, so the tuple contains both the recursive information below that predecessor and the result of `P` for the predecessor itself.
Then we can define the type of the recursion operator, where we use `Below-вЙ§` as the type of the induction hypotheses:
```
rec-вЙ§ : (P : вИА m n вЖТ m вЙ§ n вЖТ Set)
      вЖТ (вИА m n вЖТ (mвЙ§n : m вЙ§ n) вЖТ Below-вЙ§ P m n mвЙ§n вЖТ P m n mвЙ§n)
      вЖТ  вИА m n вЖТ (mвЙ§n : m вЙ§ n) вЖТ P m n mвЙ§n
```
The operational intuition of strong induction is to never forget previous results and accumulate them in a tuple instead, which is used as the strengthened induction hypotheses.
A clean way of implementing `rec-вЙ§` is to write a function constructing such tuples first:
```
below-вЙ§ : (P : вИА m n вЖТ m вЙ§ n вЖТ Set)
        вЖТ (p : вИА m n вЖТ (mвЙ§n : m вЙ§ n) вЖТ Below-вЙ§ P m n mвЙ§n вЖТ P m n mвЙ§n)
        вЖТ вИА m n вЖТ (mвЙ§n : m вЙ§ n) вЖТ Below-вЙ§ P m n mвЙ§n
below-вЙ§ P p m .m       (base refl) = tt
below-вЙ§ P p m .(suc _) (step mвЙ§n)  = let b = below-вЙ§ P p m _ mвЙ§n
                                      in (b , p m _ mвЙ§n b)
```
The type of `below-вЙ§` is the same as that of `rec-вЙ§` except that the result type is the `Below` tuple type.
In the `base` case, there are no smaller proofs, so the tuple is just `tt`.
In the `step` case, we first build the tuple `b` below `mвЙ§n` recursively (which are the strengthened hypotheses), and then extend the tuple with the result for `mвЙ§n` itself, obtained by applying the method `p` to `b` (the strengthened induction hypotheses).
Then all it takes to get from `below-вЙ§` to `rec-вЙ§` is a final application of `p`:
```
rec-вЙ§ : (P : вИА m n вЖТ m вЙ§ n вЖТ Set)
      вЖТ (вИА m n вЖТ (mвЙ§n : m вЙ§ n) вЖТ Below-вЙ§ P m n mвЙ§n вЖТ P m n mвЙ§n)
      вЖТ  вИА m n вЖТ (mвЙ§n : m вЙ§ n) вЖТ P m n mвЙ§n
rec-вЙ§ P p m n mвЙ§n = p m n mвЙ§n (below-вЙ§ P p m n mвЙ§n)
```

Same as `case-вЙ§`, we want `rec-вЙ§` to end with a general dependent function type `(ќі : ќФ) вЖТ T ќі`, so we apply Goguen et alвАЩs construction to `rec-вЙ§` and get
```
basic-rec-вЙ§ : (ќФ : Set) (T : ќФ вЖТ Set) (t : (ќі : ќФ) вЖТ ќ£[ m вИИ вДХ ] ќ£[ n вИИ вДХ ] (m вЙ§ n))
            вЖТ ((ќі : ќФ) (m n : вДХ) (mвЙ§n : m вЙ§ n) вЖТ (b : Below-вЙ§ (ќї m' n' mвЙ§n' вЖТ (ќі : ќФ) вЖТ (m' , n' , mвЙ§n') вЙ° t ќі вЖТ T ќі) m n mвЙ§n) вЖТ (m , n , mвЙ§n) вЙ° t ќі вЖТ T ќі)
            вЖТ  (ќі : ќФ) вЖТ T ќі
basic-rec-вЙ§ ќФ T t p ќі = let (m , n , mвЙ§n) = t ќі in rec-вЙ§ (ќї m' n' mвЙ§n' вЖТ (ќі : ќФ) вЖТ (m' , n' , mвЙ§n') вЙ° t ќі вЖТ T ќі) (ќї m n mвЙ§n b ќі вЖТ p ќі m n mвЙ§n b) m n mвЙ§n ќі refl
```
When we use `basic-rec-вЙ§` to write a function `(ќі : ќФ) вЖТ T ќі` that recurses on `t ќі`, first we specify `ќФ`, `T`, and `t`, and then we write a function with the same input and output but extended with additional arguments.
The first three additional arguments `m`, `n`, and `mвЙ§n` can be thought of as the results of pattern-matching `t ќі` with a tuple because there is an equality constraint `(m , n , mвЙ§n) вЙ° t ќі` near the end.
Then we get a `Below-вЙ§` argument that allows us to make recursive calls as long as the `mвЙ§n` proof gets structurally smaller.
Such a recursive call can change the input `ќі` as needed (to something larger or even unrelated), but the `mвЙ§n` part of `ќі` should get smaller for the recursive call to be legitimate.
So the `Below-вЙ§` argument contains functions of type `(ќі : ќФ) вЖТ (m' , n' , mвЙ§n') вЙ° t ќі вЖТ T ќі` for all `(m' , n' , mвЙ§n')` that are smaller than `(m , n , mвЙ§n)`.

* non-standard operators: `вДХPair`
* In general, the purpose of the basic operators is to generalise the ordinary case and recursion operators from the data being analysed to an arbitrary input context.
A basic case operator introduces the arguments required by each constructor case, together with an equality constraint relating that local constructor form to the data extracted from the global input.
A basic recursion operator introduces the recursive information available below the targeted data, again with an equality constraint relating the local recursive problem to the extracted data.

### Basic analysis translation

Once we have the basic case analysis and recursion operators, we can translate an ordinary pattern-matching definition into a program constructed using these operators. Take `wk` again.

* A general pattern: basic-rec then basic-case, so that the Below argument is a part of input to basic-case, and can be expanded by the case analysis

The function takes three inputs: `m : вДХ`, `n : вДХ`, and `m<n : suc m вЙ§ n`. In the translation, we package them into a triple `(m , n , m<n)`, which serves as the global input `ќі`.
Among these inputs, the proof `m<n` is the target to be analysed. It belongs to the indexed datatype `_вЙ§_`, the extraction function therefore returns the `_вЙ§_` data along with its surrounding indices: `(suc m , n , m<n)`.
The translation then proceeds with the basic recursion and case analysis operators for `_вЙ§_`. The outer layer is `basic-rec-вЙ§`:
```

wk m n m<n = basic-rec-вЙ§ (ќ£[ m вИИ вДХ ] ќ£[ n вИИ вДХ ] suc m вЙ§ n) (ќї (m , n , _) вЖТ m вЙ§ n) (ќї (m , n , m<n) вЖТ (suc m , n , m<n))
                         (ќї {(m , n , m<n) .(suc m) .n .m<n b refl вЖТ {!   !}})
             (m , n , m<n)
```
Applying `basic-rec-вЙ§` introduces recursive information by `Below` argument.

* Maybe reserve for case, where the connection to dependent pattern matching (in Agda) is clearer:
It also introduces an equality constraint connecting the local data being analysed with the data extracted from the global input.
This constraint is then solved by unification, so the local variables is specialised.

After the recursion layer has made the recursive information available, we apply `basic-case-вЙ§`, passing along the remaining arguments, including the recursive argument `b`. This splits the extracted proof into the two constructor cases of `_вЙ§_` (`base` and `step`):
```

                 P = ќї Dm Dn Dm<n вЖТ ((m , n , m<n) : ќ£[ m вИИ вДХ ] ќ£[ n вИИ вДХ ] suc m вЙ§ n) вЖТ (Dm , Dn , Dm<n) вЙ° (suc m , n , m<n) вЖТ m вЙ§ n
             in basic-rec-вЙ§ (ќ£[ m вИИ вДХ ] ќ£[ n вИИ вДХ ] suc m вЙ§ n) (ќї (m , n , _) вЖТ m вЙ§ n) (ќї (m , n , m<n) вЖТ (suc m , n , m<n))
                            (ќї {(m , n , m<n) .(suc m) .n .m<n b refl вЖТ
                                basic-case-вЙ§ (ќ£[ m вИИ вДХ ] ќ£[ n вИИ вДХ ] ќ£[ m<n вИИ suc m вЙ§ n ] Below-вЙ§ P (suc m) n m<n) (ќї (m , n , _) вЖТ m вЙ§ n) (ќї (m , n , m<n , _) вЖТ (suc m , n , m<n))
                                             (ќї {(m , n , m<n , b) mвВА nвВА refl refl вЖТ step (base refl)})
                                             (ќї {(m , .(suc nвВА) , .(step mвВАвЙ§nвВА) , b) mвВА nвВА mвВАвЙ§nвВА refl вЖТ let (_ , b') = b in step (b' (m , nвВА , mвВАвЙ§nвВА) refl)})
                                (m , n , m<n , b)})
                (m , n , m<n)
```
In the `base` case, the equality constraint refines the indices so that the goal can be solved directly. In the `step` case, after solving the constraint, the proof has the form `step mвВАвЙ§nвВА`. At this point the `Below` argument is expanded and provides the recursive result for the predecessor proof. Applying `step` to that recursive result gives the required proof.

* More concretely: Notice that the translation applies recursion before case analysis. This gives us the recursive information first, so that after the later case split, the recursive result is already available in the step branch.
