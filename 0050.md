## Dependently typed program transformations

(*This post is jointly written with Shu-Ting Hsu.*)

While writing Agda programs, pattern matching plays an important role in shaping the structure of a definition — that is, applying different pattern matching strategies to the same function leads to different program structures corresponding to different ways of performing case analyses.
In particular, when working with an indexed data type, a beginner may analyse the indices first but end up with a more complex program, and then they are usually taught that it can be more straightforward to analyse the data.
For example, consider this definition of `_≤_` (which is a renamed version of `_≤′_` from the standard library)
```
data _≤_ : ℕ → ℕ → Set where
  base : m ≡ n → m ≤ n
  step : m ≤ n → m ≤ suc n
```
and suppose that we want to weaken `suc m ≤ n` (that is, `m < n`) to `m ≤ n`:
```
wk : (m n : ℕ) → suc m ≤ n → m ≤ n
wk m .(suc m) (base refl)    = step (base refl)
wk m .(suc n) (step {n} m<n) = step (wk m n m<n)
```
This definition performs a case analysis on the proof `suc m ≤ n`.
To emphasise that the pattern matching on the second argument is not performed by the programmer but forced by unification, we use dot patterns explicitly (and therefore need to explicitly bind `n` in the `step` constructor in the second clause).
A beginner might be unfamiliar with the idea of induction on proofs though, and might write the following version that analyses the indices `m` and `n` first:
```
wk' : (m n : ℕ) → suc m ≤ n → m ≤ n
wk'  zero    zero          m<n         = base refl
wk'  zero   (suc .zero)    (base refl) = step (base refl)
wk'  zero   (suc n)        (step m<n)  = step (wk' zero n m<n)
wk' (suc m)  zero          (base ())
wk' (suc m) (suc .(suc m)) (base refl) = step (base refl)
wk' (suc m) (suc n)        (step m<n)  = step (wk' (suc m) n m<n)
```
Here, `wk'` is finished with six cases, and is more complex than `wk`.
In all the cases except the first one,

* once the indices `m` and `n` have been fixed, the shape of `suc m ≤ n` is already forced. (still need to do the matching explicitly)
  * the surrounding index patterns already determine which proof forms are possible
* `base refl` vs `step (base refl)` in the first case

Thus `wk` and `wk'` have different case analysis structures, even though extensionally they define the same function.
So, what is the connection between them? By transforming `wk'` step by step into `wk`, we can make those structural differences explicit.

### Transforming `wk'` to `wk`

The first step is to fully expand the two levels of case analyses in every clause.
In the case of `wk'`, only the first clause still needs to be expanded:
```
  wk₁ : (m n : ℕ) → suc m ≤ n → m ≤ n
  wk₁  zero    zero          (base _)    = base refl
  wk₁  zero   (suc .zero)    (base refl) = step (base refl)
  wk₁  zero   (suc n)        (step m<n)  = step (wk₁ zero n m<n)
  wk₁ (suc m)  zero          (base ())
  wk₁ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wk₁ (suc m) (suc n)        (step m<n)  = step (wk₁ (suc m) n m<n)
```
Next, we reorder the clauses so that the clauses with the same patterns for  `suc m ≤ n` are grouped together, to indicate that the case analysis on the indexed data `suc m ≤ n` is now performed first (although the case tree generated by Agda won’t reflect that):
```
  wk₂ : (m n : ℕ) → suc m ≤ n → m ≤ n
  wk₂  zero    zero          (base _)    = base refl
  wk₂  zero   (suc .zero)    (base refl) = step (base refl)
  wk₂ (suc m)  zero          (base ())
  wk₂ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wk₂  zero   (suc n)        (step m<n)  = step (wk₂  zero   n m<n)
  wk₂ (suc m) (suc n)        (step m<n)  = step (wk₂ (suc m) n m<n)
```
In the `step` clauses, the right-hand sides do not actually depend on whether the first argument is `zero` or `suc m` — the difference is only in the first arguments of the recursive calls, but what they both do is copying the first argument, regardless of whether that’s `zero` or `suc m`.
The separate clauses can therefore be merged into a single one:
```
  wk₃ : (m n : ℕ) → suc m ≤ n → m ≤ n
  wk₃  zero    zero          (base _)    = base refl
  wk₃  zero   (suc .zero)    (base refl) = step (base refl)
  wk₃ (suc m)  zero          (base ())
  wk₃ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wk₃ m       (suc n)        (step m<n)  = step (wk₃ m n m<n)
```
* As for the `base` clauses, the right-hand side of the second and the fourth clauses are the same, but the third clause seems to do something different.
* The third clause may be easier to handle because the domain is empty, so all definitions are extensionally equivalent; still, can we actually make it look the same with the other clauses?
* The first clause is also different, but has been mentioned above; this one seems trickier because `base refl` and `step (base refl)` look completely different.
* The transformations haven’t been finished, and we need more machinery.

### Basic analysis operators

The transformation above is a reorganization of pattern matching. To formalize it we need to make the underlying mechanism of dependent pattern matching explicit. One way to make that mechanism explicit is to translate the clause-based definition into the corresponding eliminator, as in Goguen, McBride, and McKinna’s [Eliminating Dependent Pattern Matching](https://link.springer.com/chapter/10.1007/11780274_27).

Take `_≤_` for example, the eliminator is:
```
  elim-≤ : (P : (m n : ℕ) → m ≤ n → Set)
         → ({m n : ℕ} {m≡n : m ≡ n} → P m n (base m≡n))
         → ({m n : ℕ} {m≤n : m ≤ n} → P m n m≤n → P m (suc n) (step m≤n))
         → (m n : ℕ) (m≤n : m ≤ n) → P m n m≤n
  elim-≤ P pb ps m .m       (base refl) = pb
  elim-≤ P pb ps m .(suc _) (step m≤n)  = ps (elim-≤ P pb ps m _ m≤n)
```
However, the eliminator is still too coarse. Following McBride’s presentation in [Epigram: Practical Programming with Dependent Types](https://link.springer.com/chapter/10.1007/11546382_3), the eliminator may be understood as a combination of **case analysis** and **recursion**. To study transformations of pattern matching structure more systematically, it is useful to separate them:
```
case-≤ : (P : ∀ m n → m ≤ n → Set)
       → ((m n : ℕ) (m≡n : m ≡ n) → P m n (base m≡n))
       → ((m n : ℕ) (m≤n : m ≤ n) → P m (suc n) (step m≤n))
       → (m n : ℕ) (m≤n : m ≤ n) → P m n m≤n
case-≤ P pb ps m n m≤n = elim-≤ P (λ {m₀ n₀ m₀≡n₀} → pb m₀ n₀ m₀≡n₀) (λ {m₀ n₀ m₀≤n₀} _ → ps m₀ n₀ m₀≤n₀) m n m≤n

rec-≤ : (P : ∀ m n → m ≤ n → Set)
      → (∀ m n → (m≤n : m ≤ n) → Below-≤ P m n m≤n → P m n m≤n)
      → ∀ m n → (m≤n : m ≤ n) → P m n m≤n
rec-≤ P p m n m≤n = p m n m≤n (below-≤ P p m n m≤n)
```
Separating these two operators makes the structure of the eliminator more visible. Even so, the result is still somewhat tied to the datatype being analysed: the operators are phrased directly over proofs of `_≤_`, with motives chosen specifically for that family.

In an actual dependent program, however, the object we want to inspect is often embedded in a larger input. Thus, we need a more flexible formulation of the case operator. A more general formulation is given by the basic analysis mechanism in Goguen, McBride, and McKinna’s [Eliminating Dependent Pattern Matching](https://link.springer.com/chapter/10.1007/11780274_27):
```
basic-case-≤ : (Δ : Set) (T : Δ → Set) (t : Δ → Σ[ m ∈ ℕ ] Σ[ n ∈ ℕ ] (m ≤ n))
             → ((δ : Δ) (m n : ℕ) (m≡n : m ≡ n) → (m , n , base m≡n) ≡ t δ → T δ)
             → ((δ : Δ) (m n : ℕ) (m≤n : m ≤ n) → (m , suc n , step m≤n) ≡ t δ → T δ)
             → (δ : Δ) → T δ
basic-case-≤ Δ T t m₁ m₂ δ = let (m , n , m≤n) = t δ in case-≤ (λ m n m≤n → (δ : Δ) → (m , n , m≤n) ≡ t δ → T δ) (λ m n m≡n δ → m₁ δ m n m≡n) (λ m n m≤n δ → m₂ δ m n m≤n) m n m≤n δ refl

basic-rec-≤ : (Δ : Set) (T : Δ → Set) (t : (δ : Δ) → Σ[ m ∈ ℕ ] Σ[ n ∈ ℕ ] (m ≤ n))
            → ((δ : Δ) (m n : ℕ) (m≤n : m ≤ n) → (b : Below-≤ (λ m n m≤n → (δ : Δ) → (m , n , m≤n) ≡ t δ → T δ) m n m≤n) → (m , n , m≤n) ≡ t δ → T δ)
            → (δ : Δ) → T δ
basic-rec-≤ Δ T t p δ = let (m , n , m≤n) = t δ in rec-≤ (λ m n m≤n → (δ : Δ) → (m , n , m≤n) ≡ t δ → T δ) (λ m n m≤n b δ → p δ m n m≤n b) m n m≤n δ refl
```
The generalization works by adding a global input `δ : Δ` and using `t` to extract from it the data to be analysed. The motive is then rebuilt so that the result is still stated as `T δ`, while each branch carries the equality constraint connecting the local constructor form back to the extracted data.
