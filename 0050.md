## Dependently typed program transformations

(*This post is jointly written with Shu-Ting Hsu.*)

While writing Agda programs, pattern matching plays an important role in shaping the structure of a definition тАФ that is, applying different pattern matching strategies to the same function leads to different program structures corresponding to different ways of performing case analyses.
In particular, when working with an indexed data type, a beginner may analyse the indices first but end up with a more complex program, and then they are usually taught that it can be more straightforward to analyse the data.
For example, consider this definition of `_тЙд_` (which is a renamed version of `_тЙдтА▓_` from the standard library)
```
data _тЙд_ : тДХ тЖТ тДХ тЖТ Set where
  base : m тЙб n тЖТ m тЙд n
  step : m тЙд n тЖТ m тЙд suc n
```
and suppose that we want to weaken `suc m тЙд n` (that is, `m < n`) to `m тЙд n`:
```
wk : (m n : тДХ) тЖТ suc m тЙд n тЖТ m тЙд n
wk m .(suc m) (base refl)    = step (base refl)
wk m .(suc n) (step {n} m<n) = step (wk m n m<n)
```
This definition performs a case analysis on the proof `suc m тЙд n`.
To emphasise that the pattern matching on the second argument is not performed by the programmer but forced by unification, we use dot patterns explicitly (and therefore need to explicitly bind `n` in the `step` constructor in the second clause).
A beginner might be unfamiliar with the idea of induction on proofs though, and might write the following version that analyses the indices `m` and `n` first:
```
wk' : (m n : тДХ) тЖТ suc m тЙд n тЖТ m тЙд n
wk'  zero    zero          m<n         = base refl
wk'  zero   (suc .zero)    (base refl) = step (base refl)
wk'  zero   (suc n)        (step m<n)  = step (wk' zero n m<n)
wk' (suc m)  zero          (base ())
wk' (suc m) (suc .(suc m)) (base refl) = step (base refl)
wk' (suc m) (suc n)        (step m<n)  = step (wk' (suc m) n m<n)
```
Here, `wk'` is finished with six cases, and is more complex than `wk`.
In all the cases except the first one,

* once the indices `m` and `n` have been fixed, the shape of `suc m тЙд n` is already forced. (still need to do the matching explicitly)
  * the surrounding index patterns already determine which proof forms are possible
* `base refl` vs `step (base refl)` in the first case

Thus `wk` and `wk'` have different case analysis structures, even though extensionally they define the same function.
So, what is the connection between them? By transforming `wk'` step by step into `wk`, we can make those structural differences explicit.

### Transforming `wk'` to `wk`

The first step is to fully expand the two levels of case analyses in every clause.
In the case of `wk'`, only the first clause still needs to be expanded:
```

  wkтВБ  zero    zero          (base _)    = base refl

  wkтВБ  zero   (suc n)        (step m<n)  = step (wkтВБ zero n m<n)
  wkтВБ (suc m)  zero          (base ())
  wkтВБ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wkтВБ (suc m) (suc n)        (step m<n)  = step (wkтВБ (suc m) n m<n)
```
Next, we reorder the clauses so that the clauses with the same patterns for  `suc m тЙд n` are grouped together, to indicate that the case analysis on the indexed data `suc m тЙд n` is now performed first (although the case tree generated by Agda wonтАЩt reflect that):
```

  wkтВВ  zero    zero          (base _)    = base refl
  wkтВВ  zero   (suc .zero)    (base refl) = step (base refl)
  wkтВВ (suc m)  zero          (base ())
  wkтВВ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wkтВВ  zero   (suc n)        (step m<n)  = step (wkтВВ  zero   n m<n)
  wkтВВ (suc m) (suc n)        (step m<n)  = step (wkтВВ (suc m) n m<n)
```
In the `step` clauses, the right-hand sides do not actually depend on whether the first argument is `zero` or `suc m` тАФ the difference is only in the first arguments of the recursive calls, but what they both do is copying the first argument, regardless of whether thatтАЩs `zero` or `suc m`.
The separate clauses can therefore be merged into a single one:
```

  wkтВГ  zero    zero          (base _)    = base refl
  wkтВГ  zero   (suc .zero)    (base refl) = step (base refl)
  wkтВГ (suc m)  zero          (base ())
  wkтВГ (suc m) (suc .(suc m)) (base refl) = step (base refl)
  wkтВГ m       (suc n)        (step m<n)  = step (wkтВГ m n m<n)
```
* As for the `base` clauses, the right-hand side of the second and the fourth clauses are the same, but the third clause seems to do something different.
* The third clause may be easier to handle because the domain is empty, so all definitions are extensionally equivalent; still, can we actually make it look the same with the other clauses?
* The first clause is also different, but has been mentioned above; this one seems trickier because `base refl` and `step (base refl)` look completely different.
* The transformations havenтАЩt been finished, and we need more machinery.

### Basic analysis operators

Back to the transformations from `wk'` to `wk`, to see all the steps we need to make the underlying mechanism of dependent pattern matching explicit.
One way to make that mechanism explicit is to translate the clause-based definition into the corresponding eliminators, as in Goguen, McBride, and McKinnaтАЩs [*Eliminating dependent pattern matching*](https://doi.org/10.1007/11780274_27).
For example, the eliminator for `_тЙд_` is:
```
  elim-тЙд : (P : (m n : тДХ) тЖТ m тЙд n тЖТ Set)
         тЖТ ({m n : тДХ} {mтЙбn : m тЙб n} тЖТ P m n (base mтЙбn))
         тЖТ ({m n : тДХ} {mтЙдn : m тЙд n} тЖТ P m n mтЙдn тЖТ P m (suc n) (step mтЙдn))
         тЖТ (m n : тДХ) (mтЙдn : m тЙд n) тЖТ P m n mтЙдn
  elim-тЙд P pb ps m .m       (base refl) = pb
  elim-тЙд P pb ps m .(suc _) (step mтЙдn)  = ps (elim-тЙд P pb ps m _ mтЙдn)
```
But if we look at an actual program, in particular one in Epigram, weтАЩll see that thereтАЩs some distance between the kind of program we want to write and eliminators.
For example, Section 2 of McBrideтАЩs [*Epigram: Practical programming with dependent types*](https://doi.org/10.1007/11546382_3) gave this standard example:
```
plus x y тЗР rec x {
  plus x y тЗР case x {
    plus  zero   y тЗТ y
    plus (suc x) y тЗТ suc (plus x y) } }
```
There are two things to observe here: (i) Conor chose to separate *recursion* and *case analysis*; (ii) the target to be recursed over or analysed (`x` in the example above) may be part of a larger input (`x` and `y` above).
But eliminators mix recursion and case analysis, and assume that the proof and its indices are the only things in the input тАФ for example, `elim-тЙд` ends with `(m n : тДХ) (mтЙдn : m тЙд n) тЖТ P m n mтЙдn` instead of a general dependent function type `(╬┤ : ╬Ф) тЖТ T ╬┤`.
So instead of translating pattern-matching programs directly to eliminators, Goguen et al derived separate recursion and case analysis operators from eliminators, and adapted them for translating general dependent functions.
(These operators could be thought of as giving a semantics to EpigramтАЩs `rec` and `case`.)

WeтАЩll look at case analysis first.
The following `case-тЙд` operator is the same as `elim-тЙд` except that the induction hypothesis in the `step` method is dropped, so the `case-тЙд` operator only performs a case analysis on a proof of `m тЙд n`.
```

       тЖТ ((m n : тДХ) (mтЙбn : m тЙб n) тЖТ P m n (base mтЙбn))
       тЖТ ((m n : тДХ) (mтЙдn : m тЙд n) тЖТ P m (suc n) (step mтЙдn))
       тЖТ (m n : тДХ) (mтЙдn : m тЙд n) тЖТ P m n mтЙдn

```
Like `elim-тЙд`, the type of `case-тЙд` ends with `(m n : тДХ) (mтЙдn : m тЙд n) тЖТ P m n mтЙдn`, but we want to change that to a general dependent function type `(╬┤ : ╬Ф) тЖТ T ╬┤`:
```

             тЖТ ((╬┤ : ╬Ф) (m n : тДХ) (mтЙбn : m тЙб n) тЖТ (m , n , base mтЙбn) тЙб t ╬┤ тЖТ T ╬┤)
             тЖТ ((╬┤ : ╬Ф) (m n : тДХ) (mтЙдn : m тЙд n) тЖТ (m , suc n , step mтЙдn) тЙб t ╬┤ тЖТ T ╬┤)
             тЖТ (╬┤ : ╬Ф) тЖТ T ╬┤

```
Goguen et al called such an operator тАШbasic analysisтАЩ, hence the name `basic-case-тЙд`.
The first two arguments of `basic-case-тЙд` specify what `╬Ф` and `T` are.
The next argument `t` extracts the proof and its indices from an input `╬┤ : ╬Ф` to be analysed.
In the two methods, we add equality constraints stating that the extracted proof and its indices are specialised to `base` and `step` between the input `(╬┤ : ╬Ф)` and the output `T ╬┤`.
So when we use `basic-case-тЙд` to write a function `(╬┤ : ╬Ф) тЖТ T ╬┤` that analyses `t ╬┤`, we split the programming problem into two cases; in each case, we still write a function from `╬┤ : ╬Ф` to `T ╬┤` but can assume an additional equality constraint that `t ╬┤` has a more specialised form.

* How `basic-case-тЙд` is proved from `case-тЙд`: The motive is then rebuilt so that the result is still stated as `T ╬┤`, while each branch carries the equality constraint connecting the local constructor form back to the extracted data.
* Goguen et al gave a general construction for deriving basic analysis operators from тАШelimination operatorsтАЩ (which hopefully we donтАЩt have to explain in detail)

Now, we move on to recursion.
For case analysis, the methods only need to handle the immediate constructor shape of the proof being analysed.
For recursion, however, the methods also need access to the results of recursive calls on structurally smaller proofs.

* results on *all* structurally smaller proofs тАФ strong induction
* We therefore first define a type describing what recursive information is available below тАФstructurally smaller thanтАФ a given proof.

Defined below, `Below-тЙд P m n mтЙдn` is an iterated tuple type asserting that the predicate `P` holds for every value structurally smaller than the current proof `mтЙдn`:
```

```
In the `base` case, there is no smaller proof, so the tuple is trivial.
In the `step` case, the proof contains a predecessor proof `mтЙдn`, so the tuple contains both the recursive information below that predecessor and the result of `P` for the predecessor itself.
Then we can define the type of the recursion operator, where we use `Below-тЙд` as the type of the induction hypotheses:
```

      тЖТ (тИА m n тЖТ (mтЙдn : m тЙд n) тЖТ Below-тЙд P m n mтЙдn тЖТ P m n mтЙдn)
      тЖТ  тИА m n тЖТ (mтЙдn : m тЙд n) тЖТ P m n mтЙдn
```
The operational intuition of strong induction is to never forget previous results and accumulate them in a tuple instead, which is used as the strengthened induction hypotheses.
A clean way of implementing `rec-тЙд` is to write a function constructing such tuples first:
```

        тЖТ (p : тИА m n тЖТ (mтЙдn : m тЙд n) тЖТ Below-тЙд P m n mтЙдn тЖТ P m n mтЙдn)
        тЖТ тИА m n тЖТ (mтЙдn : m тЙд n) тЖТ Below-тЙд P m n mтЙдn

below-тЙд P p m .(suc _) (step mтЙдn)  = let b = below-тЙд P p m _ mтЙдn
                                      in (b , p m _ mтЙдn b)
```
The type of `below-тЙд` is the same as that of `rec-тЙд` except that the result type is the `Below` tuple type.
In the `base` case, there are no smaller proofs, so the tuple is just `tt`.
In the `step` case, we first build the tuple `b` below `mтЙдn` recursively (which are the strengthened hypotheses), and then extend the tuple with the result for `mтЙдn` itself, obtained by applying the method `p` to `b` (the strengthened induction hypotheses).
Then all it takes to get from `below-тЙд` to `rec-тЙд` is a final application of `p`:
```

      тЖТ (тИА m n тЖТ (mтЙдn : m тЙд n) тЖТ Below-тЙд P m n mтЙдn тЖТ P m n mтЙдn)
      тЖТ  тИА m n тЖТ (mтЙдn : m тЙд n) тЖТ P m n mтЙдn

```

Same as `case-тЙд`, we want `rec-тЙд` to end with a general dependent function type `(╬┤ : ╬Ф) тЖТ T ╬┤`, so we apply Goguen et alтАЩs construction to `rec-тЙд` and get
```

            тЖТ ((╬┤ : ╬Ф) (m n : тДХ) (mтЙдn : m тЙд n) тЖТ (b : Below-тЙд (╬╗ m' n' mтЙдn' тЖТ (╬┤ : ╬Ф) тЖТ (m' , n' , mтЙдn') тЙб t ╬┤ тЖТ T ╬┤) m n mтЙдn) тЖТ (m , n , mтЙдn) тЙб t ╬┤ тЖТ T ╬┤)
```

When we use `basic-rec-тЙд` to write a function `(╬┤ : ╬Ф) тЖТ T ╬┤` that recurses on `t ╬┤`, first we specify `╬Ф`, `T`, and `t`, and then we write a function with the same input and output but extended with additional arguments.
The first three additional arguments `m`, `n`, and `mтЙдn` can be thought of as the results of pattern-matching `t ╬┤` with a tuple because there is an equality constraint `(m , n , mтЙдn) тЙб t ╬┤` near the end.
Then we get a `Below-тЙд` argument that allows us to make recursive calls as long as the `mтЙдn` proof gets structurally smaller.
Such a recursive call can change the input `╬┤` as needed (to something larger or even unrelated), but the `mтЙдn` part of `╬┤` should get smaller for the recursive call to be legitimate.
So the `Below-тЙд` argument contains functions of type `(╬┤ : ╬Ф) тЖТ (m' , n' , mтЙдn') тЙб t ╬┤ тЖТ T ╬┤` for all `(m' , n' , mтЙдn')` that are smaller than `(m , n , mтЙдn)`.

Back to the question at the beginning of this section, we can now explain the situation more precisely:
1. The redundant-case elimination in the `step` branches succeeds only accidentally in the clause-based version. Since Agda automatically omits the impossible branches, we only see the two surviving `step` clauses, and these can be merged after reordering. In the basic-analysis version, the omitted branches reappear as contradiction branches, making clear that the clause-based transformation was relying on AgdaтАЩs omission of impossible cases.
2. The basic-analysis translation lets us split case analysis and unification, which are combined into a single step in ordinary pattern matching. This is the crucial technique for completing the transformation: by delaying the equality constraints, we can control where unification takes place. The unification stategy decides how the right-hand side of the program can be written, and therefore can perform different definitions of the same function.
3. Apart from unification, the two basic-analysis transformation laws stated above correspond to case-reordering and rdundant-case elimination in clause-based transformation, which determine only the outer structure of the program transformation but not how the branch bodies are constructed.

One limitation should be noted here.
These laws hold at the level of basic analysis operators, not directly at the surface pattern matching level.
This is especially important for the delayed-constraint transformation: ordinary pattern matching does not expose equality constraints separately from unification, so this step cannot be stated cleanly at the clause level.
